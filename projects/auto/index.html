Autonomous Drone Navigation using 3D Gaussian Splatting and Foundation Models

An end-to-end autonomous drone navigation system combining 3D Gaussian Splatting, DINOv2 for Visual Place Recognition, and hybrid RRT*/APF planning.
16 min · Lav-niya

Traditional drone navigation relies heavily on sparse point clouds and GPS. But what if a drone could understand its environment exactly like a human does, through photorealistic memory and real-time visual recognition?

In this project, I built an end-to-end autonomous drone stack in AirSim. Instead of standard SLAM, this system uses 3DGS to build a rich offline map. It then leverages DINOv2 (AnyLoc) to create a visual memory bank, allowing the drone to wake up blind, look around, and instantly localize itself. Finally, a hybrid RRT* planning engine and a Dynamic Artificial Potential Field (APF) guide the drone to its target while actively dodging moving obstacles.

The Tech Stack

  • Simulation: Microsoft AirSim
  • Mapping: 3DGS, Open3D, SciPy(Occupancy Grids)
  • Visual Place Recognition (VPR): PyTorch, DINOv2 (AnyLoc), FAISS Vector Database, Kornia (DISK + LightGlue)
  • Pathfinding: RRT*, B-Spline Smoothing
  • Obstacle Avoidance: DBSCAN Clustering, Dynamic Artificial Potential Field (APF)

Phase 1: The Memory Bank and Voxel grid#

Before drone can fly autonomously, it needs two things: a physical understanding of the world to avoid crashing and a visual memory to know where it is without GPS.

From Gaussian Splats to Collision Grids

Why 3DGS? Traditional photogrammetry generates point clouds and textured meshes. While great for collision avoidance, meshes often produce “tearing” or artifacts when rendering novel views. 3DGS represents the scene as continuous, translucent mathematical ellipsoids. This allows for photorealistic novel view synthesis which is absolutely critical for generating accurate synthetic training data for the drone’s vision system.

First, I extracted the raw point cloud from 3DGS model. To extract the true base colors of the scene, I decoded the zeroth degree sphereical harmonics $(S H_0)$: $$C_{RGB} = SH_0 \times 0.282094 + 0.5$$

Once extracted and rotated into a standard gravity-aligned coordinate system, the continuous points are quantized into a discrete 3D occupancy matrix (a voxel grid) using a 0.2m resolution. To map a physical continuous coordinate (P) to a discrete grid index, I applied:

$$Index = \left( \frac{P - P_{min}}{V_size}\right)$$

The Safety buffer: Drones have physical wingspans. To prevent the center of the drone from pahtfinding too close to a wall, I applied a 3D binary dialation using scipy.ndimage. This “inflates” the walls of the voxel grid by 0.5 meters, creating an invisible safety buffer.

Visual Place Recognition: The AnyLoc method When the drone spawns, it is completely blind. It takes panoramic photos of its surroundings and must mathematically deduce its X, Y, Z coordinates.

Why DINOv2 + AnyLoc over traditional SLAM or NetVLAD? Traditional local feature matchers (Like SIFT or ORB) are highly susceptible to lighting changes, shadows, and seasonal shifts. NetVLAD is robust but requires heavey, domain-specific training. I utilized the AnyLoc method, leveraging Meta’s DINOv2 vision model. Because DINOv2 understands deep semantic structures, it provides zero-shot robustness. It recognizes a building based on its structural essence, regardless of the simulator’s lighting or sun angle.

To build the database, I programmed a virtual “lawnmower” path through 3DGS model rendering thousands of synthetic images a tvarious coordinates and altitudes.

Each image is passed through DINOv2 to extract patch tokens. Instead of just taking the average of these tokens, I applied Generalized Mean (GeM) Pooling:

$$f = \left( \frac{1}{N} \sum_{i=1}^{N} x^{p}_{i} \right)^{\frac{1}{p}} $$

average pooling (p=1) dilutes distinct visual landmarks, while Max pooling $(p \rightarrow \infty)$ is too harsh and loses spatial context. By setting $p=3.0$, GeM pooling perfectly bridges this gap, it focuses the neural network’s attention on distinct structural landamarks (like the corner of a roof or a specific window) while retaining enough context of the surrounding wall.

# Extract DINOv2 patch tokens
ret = dinov2.forward_features(img_tensor)
patches = ret['x_norm_patchtokens']

# Apply Generalized Mean (GeM) Pooling (p=3.0)
patches_pow = torch.pow(patches.clamp(min=1e-6), p)
gem_pool = torch.mean(patches_pow, dim=1)
anyloc_features = torch.pow(gem_pool, 1.0/p)

# L2 Normalize for the FAISS Index
embedding = torch.nn.functional.normalize(anyloc_features, p=2, dim=1)

The FAISS Vector Database The resulting 384-dimensional GeM embeddings are fed into a FAISS IndexFlatIPS (inner product) vector database. Because I explicitly L2-normalized the feature vectors in PyTorch before indexing them, the FAISS inner product mathematically computes the Cosine Similarity.

$$ S_{C}(A,B) = \frac{A . B}{\parallel A \parallel \parallel B \parallel} $$

This approach allows the drone to compare a live, real-time photo against thousands of synthetic memory bank photos in milliseconds, instantly finding its coarse neighborhood in the map.

Phase 2: Live Navigation & Vision System#

When the drone spawns into the simulation, it is completely blind. It has no GPS and no initial starting coordinates. It must “wake up” look around, and mathematically deduce its location in the 3D map using a Coarse-to-Fine Visual Place Recognition (VPR) pipeline.

The Coarse-to-Fine Pipeline (DISK + LightGlue) The drone rotates and snaps siz panoramic images. Each image is processed through the VPR system.

  1. Coarse Search: The live image is passed through DINOv2. The resulting GeM-pooled embedding is queried against the offline FAISS Vector Database, retrieving the top 3 nearest synthetic memory bank images.

  2. Fine Verification: Coarse image retrieval is not geometrically exact enough for a drone. To pinpoint the exact location, I pass the live image and the top synthetic candidates through DISK (a local feature extractor) and LightGlue (state of the art neural matcher).

Why DISK and LightGlue? LightGlue explicitly rejects unmatchable keypoints and returns a highly accurate geometric verification (inlier count). If the inlier count is below 100, the drone rejects the view as too vague.

The Rosetta Stone: Rigid Body Transformations Once a synthetic image is verified, the drone knows where the camera was in the map’s coordinate system. But the drone flies in AirSim’s physical coordinate system (NED: North-East-Down). I built a manual Groud Control Point (GCP) anchoring script to calculate the scale and translation between these two worlds.

To prevent the map from accidently sloping or tilting, I implemented a “Flat Gravity Anchor”. By mathematically forcing the middle row of the rotation matrix to zero out pitch and roll, the drone locks onto yaw difference $(\theta)$ strictly on the XZ plane:

$$ R_{upright} = \begin{bmatrix} \cos(\theta) & 0 & -\sin(\theta) \\ 0 & 1 & 0 \\ \sin(\theta) & 0 & \cos(\theta) \end{bmatrix} $$

Sensor Fusion: Weighted Consensus Because the drone takes six photos, it might find multiple succesful localization. Instead of simple average, the drone uses the LightGlue inlier count (W) as a confidence weight to calculate its final starting coordinate:

$$ X_{final} = \frac{\sum (X_{i} \times W_{i})}{\sum W_{i}} $$

A camera starting at a highly recongnizable building (500 inliers) mathematically overpowers a camera staring at a blank path of grass (105 inliers).

Human-in-the-loop: Open3D Tactical Map Before generating a flight path, the system opens a real-time Open3D tactical interface. The operator uses W/A/S/D/E/Q keys to position a blue sphere in 3D space. To prevent user error, a real time keyCallback runs the coordinates through physical_to_voxel grid. If the user places the target inside a building, the sphere turns RED and physically locks out the Spacebar authorization key. Only when the sphere is in sage airspace (Blue) can the flight plan be executed.

Global Tactical Plannar#

Once the drone knows where it is, it needs to plot a course through the 3D voxel grid. For exploring wide-open 3D airspace rapidly, a standard grid-based searches (like A*) check far too many empty nodes, and standard RRT algorithms create highly suboptimal, wandering paths.

To solve this, I implemented RRT* (RRT-Star), an asymptotically optimal sampling-based plannar.

Tree Rewiring and Cost Optimization Unlike standard RRT, which locks in the first path it finds, RRT* actively optimizes the flight path distance during the search phase.

Whenever a new 3D node is sampled and added to the tree, the algorithm searches within a designated radius for neighboring nodes. It calculates the Eculidean cost to route through the new node. If the new route is cheaper (shorter) than a neighbor’s current path to the start, the algorithm mathematically rewires the tree.

This dynamic rewiring acts as an algorithmic string-puller, straightening out the jagged branches of the search tree in real-time before the final path is even selected.

# Check if the newly sampled node offers a cheaper path for its neighbors
for near_node in near_nodes:
    potential_cost = new_node.cost + get_distance(new_node, near_node)
    
    if potential_cost < near_node.cost:
        # Cast a 3D ray to ensure the new optimized shortcut doesn't clip a building
        if not check_collision_3d(new_node, near_node, grid):
            near_node.parent = new_node
            near_node.cost = potential_cost

3D Raycast Collision Checking To ensure branches don’t intersect the collision grid, I built a custom check_collison_3D function. It calculates the exact 3D distance between two nodes and samples points along the connecting vector at 0.5 voxel intervals. This guarantees the algorithm never accidentally jumps over thin walls or power lines.

B-Spline Trajectory Smoothing: While RRT* guarantees the shortest distance, the path still consists of straight lines and sharp corners. To translate this into a realistic quadcopter trajectory, the system applies three layers of smoothing:

  1. String Pulling: Raycasting between distant waypoints to delete redundant intermediate nodes.
  2. “Pigtail” Filter: A custom spatial filter that removes clustered waypoints (distance<0,8m) to prevent the math from creating physical loops or overshoots.
  3. Quadratic B-Splines: Using scipy.interpolate.splprep (k=2) to bend the remaining sharp corners into a continuously smooth, aerodynamic flight curve.

Local Reflexes: Dynamic Obstacle Avoidance#

A global path is useless if a dynamic obstacle moves in fromt of the drone. I built a real-time Dynamic Artificial Potential Field (APF) that runs asynchronously alongside the main flight loop.

LiDAR Voxel Downsampling and DBSCAN Running clustering algorithms on 100000 raw LiDAR points per second will crash a Python control loop. To optimize this, the raw point cloud is first quantized into a 0.5m discrete grid. By keeping only one point per voxel block, the data footprint is reduced by 90% without losing spatial boundaries. This ultra-lightweight point cloud is then fed into DBSCAN to calculate the physical centroids of incoming threats.

Ego-Motion Compensation and Time-to-Collision (TTC) Kinematics A classic problem in obstacle avoidance is ego-motion: if a drone flies forward at 5m/s, every static tree appears to be flying toward the sensor at 5m/s. To prevent the APF from panic-dodging static walls, the drone must mathematically cancel out its own movement.

I implemented rigid body kinematic to isolate true moving threats.

  1. The system reads the drone’s high speed odometry (linear and angular velocity).
  2. It rotates the world velocity into the LiDAR’s body frame using the drone’s current yaw.
  3. It calculates the false speed caused by the drone’s rotation using the cross product: $v_{rot}=w /times r$.
  4. It subtracts the drone’s ego-motion from the obstacle’s apparent velocity.

If the resulting true velocity is near zero, the object is static and is ignored. If the object is truly moving toward the drone, the APF scales the repulsive force ($F_{repulsive}$) inversely by the Time-To-Collision (TTC).

$$ F_{repulsive} = \frac{K_{repulsive}}{d \times TTC} $$

This means the drone smoothly glides past static buildings, but will violently brake and snap away from a moving target (like another drone).

# Grab drone kinematics and rotate World Velocity into the LiDAR's Body Frame
v_body_x = v_world.x_val * math.cos(yaw) + v_world.y_val * math.sin(yaw)
v_body_y = -v_world.x_val * math.sin(yaw) + v_world.y_val * math.cos(yaw)
drone_linear_vel = np.array([v_body_x, v_body_y, v_world.z_val])

# Calculate false speed caused by drone rotation (v = w x r)
drone_angular_vel = np.array([w_body.x_val, w_body.y_val, w_body.z_val])
rotational_offset = np.cross(drone_angular_vel, curr_centroid)

# Cancel out ego-motion to find the TRUE object velocity
safe_dt = max(dt, 0.05) # Guardrail against divide-by-zero explosions
apparent_velocity = (curr_centroid - last_centroid) / safe_dt
true_obj_vel = apparent_velocity + drone_linear_vel + rotational_offset

dist = np.linalg.norm(curr_centroid)

# Ignore distant objects (>10m) and static objects (true speed < 2.0m/s)
if dist <= 10.0 and np.linalg.norm(true_obj_vel) >= 2.0:
    
    # Threat is truly moving Execute Dynamic TTC Math
    dir_to_drone = -curr_centroid / dist
    closing_speed = np.dot(apparent_velocity, dir_to_drone)
    
    if closing_speed > 0.1: 
        ttc = dist / closing_speed
        if ttc < TTC_THRESHOLD:
            # Scale repulsive force inversely by Time-To-Collision
            force_dir = -curr_centroid / dist 
            force_mag = REPULSIVE_CONSTANT / (dist * ttc)
            total_repulsive_force += force_dir * force_mag

Results, Ground Truth Benchmarking and Future Work#

Because this was built in AirSim, I was able to query the Unreal Engine physics simulator for the exact real-world (X, Y, Z) coordinates of the drone at the exact moment the vision system made its guess. By translating the Neural Map coordinates into AirSim Physics coordinates, I calculated the absolute 3D Euclidean error distance.

Performance Highlights:

  • Localization accuracay: The DISK + LightGlue consensus consistently localized the drone with sub-meter accuracy (often within 0.3m to 0.8m of the unreal engine ground truth), even when spawned in visually ambiguous areas.
  • Path Planning Speed: RRT-Connect generated 3D voxel paths spanning 40+ meters through complex building layouts in under 1.5 seconds.
  • Evasive Maneuvers: The Dynamic APF successfully overrode the global trajectory, violently braking the drone to avoid simulated physics boxed dropped directly onto the flight path, before smoothly returning to the B-spline curve.

Real-World Edge Deployment (Future architecture) While this project runs monolithically alongside a simulator, the system was designed with physical “Edge AI” deployment in mind. To move this to a real drone, the UI and the “Brain” must be decoupled into a client-Server model.

  1. Edge Server (The Drone): The heavy computations (DINOv2, FAISS, RRT-Connect, APF) would be containerized via Docker and deployed onto an onboard companion computer. For high-end performance, an NVIDIA Jetson Orin Nano using TensorRT (FP16) provides robust CUDA acceleration. However, the architecture is flexible; for a budget-friendly build, edge-inference can be offloaded to cheaper alternatives like an Orange Pi 5 (RK3588 NPU) or a Raspberry Pi 5 paired with a Hailo-8 or Google Coral AI accelerator at a fraction of the cost.
  2. Client UI (Ground Control Station): The Open3D Tactical Map would run on an operator’s laptop or tablet (e.g., via ROS2 RViz or a web dashboard).
  3. The operator selects the Goal Sphere on the tablet, and a lightweight JSON payload of the (X, Y, Z) coordinates is sent to the drone via 4F LTE or RF telemetry. The drone executes the mission autonomously, relying strictly on its onboard sensros to survive.
